## 이 기록을 읽으면 바로 알아야 하는 것

현재 작업은 Savingio 공개 글 재작성 작업이 아니라, `/admin/` 안의 **Savingio Admin HQ 콘텐츠 운영센터**를 실제 운영도구로 만드는 작업이다.

이번 단계의 핵심 목표는 단순히 `중복 100%`를 표시하는 것이 아니다.

> 중복 글을 그룹으로 묶고, 대표로 남길 글·통합할 글·삭제할 글을 운영자가 한 화면에서 판단하고 실행 준비까지 할 수 있게 만드는 것.

장기 원칙은 다음과 같다.

> `검색의도 하나 = 대표글 하나`

새 글을 만들기 전에 기존 글과 중복 여부를 먼저 검사하고, 기존 대표글이 있으면 신규 작성보다 기존 글 업데이트·통합·삭제 정리를 우선한다.

# 이번 작업: Duplicate Center

## 1. 처음 발견된 문제

기존 중복 열은 아래처럼 숫자만 표시했다.

```text
중복 100%
```

이 방식으로는 운영자가 다음을 알 수 없었다.

- 무엇과 중복인지
- 어느 URL을 남겨야 하는지
- 무엇을 통합해야 하는지
- 무엇을 삭제해야 하는지
- 왜 그렇게 판단했는지

따라서 숫자 표시를 중복 그룹 중심 운영 화면으로 변경했다.

---

## 2. 구현 완료된 중복 그룹 기능

관련 파일:

- `admin/duplicate-center.js`
- `admin/duplicate-center.css`
- `admin/content-center-runtime.js`
- `functions/api/admin/content-action.js`

구현 내용:

### 중복 그룹 자동 생성

전체 Doctor 검사 후 제목과 slug의 토큰 유사도를 계산하여 중복 후보를 그룹으로 묶는다.

예시:

```text
DUP-002
통신비 할인 총정리
3개 URL
```

### 글별 운영 추천

그룹 안에서 다음 세 가지로 자동 추천한다.

- `대표 유지`
- `통합 후보`
- `삭제 후보`

대표글 선정은 현재 구현 기준으로 다음을 우선한다.

1. 품질점수
2. 본문 분량

현재는 제목·slug 기반 그룹 판정이다. 실제 본문 의미 유사도, Search Console 유입, 색인 상태, 내부링크 수를 포함하는 완전한 AI 중복 판정은 아직 아니다.

### 중복 비교창

중복 배지를 누르면 다음 내용을 보여준다.

- 그룹 번호
- 검색의도 대표 제목
- 그룹 내 URL 수
- 대표 유지 추천 글
- 각 글 제목과 URL
- 대표글과의 유사도
- 품질점수
- 본문 분량
- 운영 판단 선택상자

### 중복센터 요약

운영센터 상단에 다음 카운트를 표시한다.

- 중복 그룹
- 통합 후보
- 삭제 후보
- 검토 완료

### 그룹 검토 상태 저장

운영자가 그룹별로:

- 검토 완료
- 판단 보류

상태를 저장할 수 있다.

현재 이 검토 상태와 개별 판단은 브라우저 `localStorage`에 저장된다.

### 새 글 중복 사전검사

새 주제를 입력하면 기존 글과 비교하여 다음 중 하나를 표시한다.

```text
신규 작성 가능
```

또는

```text
새 글 생성 보류
기존 대표 후보 존재
기존 글 업데이트 또는 중복 그룹 검토 우선
```

## 3. 사용자가 실제 화면에서 확인한 상태

사용자가 `/admin/`에서 실제 Duplicate Center 화면을 열어 확인했다.

확인된 화면 예시:

- `DUP-002 중복 비교`
- 검색의도 대표: `통신비 할인 총정리`
- 3개 URL 표시
- 대표글 1개
- 통합 후보 2개
- 품질 67점
- 본문 3,519자
- 운영 판단 선택상자

즉, 중복 그룹 UI와 비교표가 Production 화면에 실제 표시되는 것까지 사용자 스크린샷으로 확인했다.

---

## 4. 사용자가 지적한 다음 문제

비교 화면에는 운영 판단 선택상자만 있었고 실제 작업 버튼이 없었다.

사용자 요구:

> 이 화면에서 통합이나 삭제 버튼도 있어야 한다.

따라서 각 행에 실행 기능을 추가했다.

## 5. 이번에 추가한 통합·삭제 준비 기능

### 화면 버튼

대표글:

```text
🔒 대표글
```

통합·삭제 대상 글:

```text
통합 준비
삭제 준비
```

선택상자의 운영 판단에 따라 버튼 문구와 동작이 바뀐다.

### 통합 준비

현재 구현된 흐름:

1. 대표글 확인
2. 통합 대상 글 확인
3. 사용자 확인창 표시
4. 대표글 원본 HTML 읽기
5. 대상 글 원본 HTML 읽기
6. 두 원본을 KV에 백업
7. 통합 검토 작업 생성
8. 대표 URL과 대상 URL 저장
9. 상태를 `duplicate_merge_review`로 저장

중요:

- 현재 단계에서는 실제 본문 병합을 수행하지 않는다.
- 대상 글을 실제 삭제하지 않는다.
- 대표글을 실제 덮어쓰지 않는다.
- 안전하게 백업과 검토 작업 등록까지만 한다.

### 삭제 준비

현재 구현된 흐름:

1. 삭제 대상 확인
2. 연결할 대표 URL 확인
3. 사용자 확인창 표시
4. 대표글 원본 HTML 읽기
5. 삭제 대상 원본 HTML 읽기
6. 두 원본을 KV에 백업
7. 삭제 검토 작업 생성
8. 대표 URL과 삭제 대상 URL 저장
9. 상태를 `duplicate_delete_review`로 저장

중요:

- 실제 파일 삭제는 아직 하지 않는다.
- Redirect 생성도 아직 하지 않는다.
- 사이트맵 제거도 아직 하지 않는다.
- 내부링크 교체도 아직 하지 않는다.
- 최종 승인 전에는 파괴적 작업을 하지 않는 구조다.

# 절대 안전 규칙

1. 중복 추천만으로 즉시 삭제하지 않는다.
2. 원본 백업 없이 통합·삭제하지 않는다.
3. 대표 URL을 확인하지 않고 삭제하지 않는다.
4. 내부링크 영향을 확인하지 않고 삭제하지 않는다.
5. 사이트맵과 검색트리에서 제거하기 전에 실제 파일만 먼저 삭제하지 않는다.
6. Cloudflare 배포와 실제 URL 검증 전에는 완료 또는 PASS로 표시하지 않는다.
7. 자동 추천이 틀릴 수 있으므로 최종 결정은 운영자 승인 후 수행한다.
8. 공식 Publisher ID `pub-7605193583747751` LOCK은 계속 유지한다.
9. GitHub `main`이 실제 구현 상태의 최종 기준이다.
