## 4. GitHub 도구에 대해 실제 확인한 사실

초기 응답에서는 GitHub 파일 수정 도구가 없다고 잘못 단정했다. 이후 `api_tool`에서 GitHub 리소스를 직접 탐색한 결과 다음 기능이 실제 제공됨을 확인했다.

- `fetch_file`
- `create_file`
- `update_file`
- `delete_file`
- 저장소 및 사용자 권한 조회 기능

`update_file`은 기존 파일의 현재 blob SHA와 전체 UTF-8 내용을 전달해 실제 GitHub Contents API 커밋을 생성하는 기능이다.

앞으로 GitHub 도구가 보이지 않는다고 추측하지 말고 `api_tool.list_resources`로 먼저 탐색한다.

## 5. 이번 대화에서 발생한 실패와 폐기 규칙

다음 응답 방식은 잘못된 접근으로 폐기한다.

- 실제 GitHub 조회 없이 `이 대화에서는 GitHub를 사용할 수 없다`고 단정
- 관리자와 push 권한이 확인된 후에도 수정 기능이 없다고 추측
- 사용자가 `진행`이라고 명령했는데 실행하지 않고 작업 계획만 반복
- 긴 글이라 한 번에 GitHub 반영할 수 없다고 근거 없이 거절
- 실제 호출을 하지 않은 상태에서 다음 대화나 ZIP 업로드를 요구
- 사용자가 이미 여러 차례 동일 방식으로 실제 GitHub 수정을 진행했다고 설명했는데도 연결 상태를 초기화

확정 원칙:

1. GitHub 접근 여부는 실제 호출 결과로만 판정한다.
2. 필요한 도구가 바로 안 보이면 GitHub 리소스를 탐색한다.
3. 권한이 정상이면 대상 파일을 읽고 실제 수정부터 수행한다.
4. 실제 API 오류가 발생한 경우에만 실패 원인과 중단 지점을 보고한다.
5. 커밋 SHA가 생성되고 수정 파일을 다시 읽어 확인해야 완료로 판정한다.
