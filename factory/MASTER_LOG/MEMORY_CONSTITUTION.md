# SAVINGIO MEMORY CONSTITUTION

최종 갱신: 2026-07-18 KST
상태: LOCK / 불변 우선 규칙
적용 대상: Savingio Factory를 이어서 작업하는 모든 대화와 모든 작업자

## 1. 최우선 목적

사용자가 과거 결정과 현재 시작점을 다시 설명하거나 증명하게 만들지 않는다.
새 대화가 시작돼도 저장소에서 현재 상태를 스스로 복구한 뒤 바로 이어서 작업한다.

## 2. 근거 우선순위

충돌이 발생하면 아래 순서를 강제한다.

1. 이 문서와 별도 헌법·LOCK 문서
2. GitHub `main`의 실제 파일과 최신 커밋
3. `factory/MASTER_LOG/MASTER_LOG_CURRENT.md`
4. `factory/MASTER_LOG/04_HANDOVER/NEXT_TASK.md`
5. 분류형 MASTER LOG 전체
6. 기존 `MASTER_LOG_PART1.md`, `MASTER_LOG_PART1-1.md`, `MASTER_LOG_PART2.md`, `MASTER_LOG_PART3.md`
7. 현재 대화의 최신 사용자 지시
8. 과거 대화 기억과 추정

상위 근거와 하위 근거가 다르면 하위 근거를 폐기한다.
기억과 저장소가 다르면 기억이 틀린 것으로 처리한다.

## 3. 현재 작업 방식 LOCK

- 공식 본집은 GitHub 저장소 `yusun7749-art/savingio`다.
- 공식 작업 기준 브랜치는 `main`이다.
- 작업은 GitHub 파일 수정 → 검증 → 커밋/PR → `main` 병합 순서로 진행한다.
- Cloudflare는 `main` 반영을 기준으로 자동 배포된다.
- ZIP은 공식 작업 원본이 아니다. 사용자가 명시적으로 요청하거나 백업·전달이 필요할 때만 만든다.
- 근거 없이 ZIP 업로드를 다시 요구하거나 ZIP 중심 작업으로 회귀하지 않는다.
- GitHub Connector로 실제 파일 생성·수정이 가능한 상태를 근거 없이 부정하지 않는다.

## 4. 새 대화 강제 부팅 절차

Savingio 작업 요청을 받으면 답변이나 개발 전에 반드시 다음을 수행한다.

1. `main` 최신 커밋 SHA와 메시지를 확인한다.
2. 이 `MEMORY_CONSTITUTION.md`와 `BOOT_ORDER.json`을 읽는다.
3. `factory/MASTER_LOG/README.md`의 읽기 순서를 확인한다.
4. `MASTER_LOG_CURRENT.md`와 `04_HANDOVER/NEXT_TASK.md`를 읽는다.
5. `01_SUCCESS`부터 `06_QA`까지 모든 분류형 MASTER LOG 문서를 빠짐없이 확인한다.
6. 기존 공식 PART 로그도 모두 확인한다.
7. 로그 기준 커밋과 최신 `main`을 비교한다.
8. 현재 실패 항목, 미완료 항목, 다음 작업을 실제 파일과 대조한다.
9. 로그가 실제 구현보다 뒤처졌다면 먼저 로그를 복구한다.
10. 현재 시작점을 내부 확정한 뒤 즉시 실행한다.

이 절차를 완료하지 않은 상태에서는 다음을 금지한다.

- 예전 ZIP 방식 안내
- 이미 완료된 질문 반복
- 연결·권한을 근거 없이 없다고 판단
- 과거 버전을 현재 상태로 안내
- 검증하지 않은 PASS 출력
- 사용자가 과거 자료를 다시 찾도록 요구

## 5. 사용자 부담 금지

다음 행동은 오류로 판정한다.

- 사용자가 이미 만든 헌법·LOCK·로그를 다시 설명하게 하는 것
- 사용자가 자신의 기억이 맞는지 과거 대화를 뒤지게 하는 것
- 현재 시작점을 찾는 책임을 사용자에게 넘기는 것
- 저장소 확인 없이 이전 작업 방식을 제안하는 것

불확실할 때는 사용자의 기억을 의심하지 말고 저장소와 로그를 먼저 확인한다.

## 6. 작업 완료 기록 규칙

실제 변경을 마친 회차에는 같은 회차에 즉시 분류 저장한다.

- 완료·성공·릴리스 → `01_SUCCESS/`
- 실패·원인·재발 금지 → `02_FAILURE/`
- 구조·작업공간·엔진·브랜드 결정 → `03_DECISIONS/`
- 다음 작업·세션 인수인계·필수 맥락 → `04_HANDOVER/`
- 중요 대화·채택 아이디어·참고자료 → `05_KNOWLEDGE/`
- 버그·회귀·검수표 → `06_QA/`

그리고 같은 회차에 반드시 갱신한다.

- `MASTER_LOG_CURRENT.md`
- `MASTER_LOG_HISTORY.md`
- `04_HANDOVER/NEXT_TASK.md`
- `04_HANDOVER/SESSION_SUMMARY.md`
- 관련 PART 로그
- 최신 커밋 또는 PR 정보
- 실제 완료 항목
- 검증 실패 및 남은 작업
- 정확한 다음 시작점

계획과 실제 완료를 섞지 않는다.
실행하지 않은 작업은 완료로 기록하지 않는다.
하나의 실행이 끝났는데 분류 저장이 끝나지 않았다면 그 실행은 종료된 것으로 보지 않는다.

## 7. 기억 복구 실패 시 처리

현재 상태가 불명확하면 추측해서 진행하지 않는다.
다만 사용자에게 과거 자료를 요구하기 전에 다음을 끝까지 수행한다.

1. 공식 로그 전체 확인
2. 최근 커밋 확인
3. 관련 파일 검색
4. 최근 PR 및 변경 파일 확인
5. 실제 배포 파일과 엔진 파일 대조

위 절차 이후에도 저장소에 근거가 없을 때만 부족한 한 가지 정보를 사용자에게 묻는다.

## 8. 고정 재시작 문장 해석

사용자가 아래와 비슷하게 말하면 즉시 전체 MASTER LOG 강제 부팅 절차를 수행한다.

- `진행`
- `시작`
- `0`
- `마스터로그보고와`
- `마스터로그 보고 와`
- `마스터로그 확인하고 와`
- `이어서 진행해`
- `Savingio Factory 이어서 시작`

특히 `마스터로그보고와`는 Savingio의 헌법, 현재 로그, 분류형 로그, 기존 PART 로그, 최신 `main`을 전부 확인하고 현재 시작점을 스스로 확정하라는 명령이다.
이 명령은 설명 요청이 아니라 상태 복구 후 실제 작업을 시작하라는 뜻이다.

## 9. 현재 공식 선언

Savingio의 기억은 대화 기억이 아니라 GitHub의 헌법, 실제 `main`, MASTER LOG로 유지한다.
사용자는 기억 복구 담당자가 아니다.
작업자가 저장소를 읽고 현재 위치를 복구한다.

## 10. 분류형 MASTER LOG LOCK

`factory/MASTER_LOG/`는 시간순 기록만 쌓는 폴더가 아니라 실행 가능한 프로젝트 기억 체계다.
모든 실행 종료 시 결과를 용도별 문서에 바로 분류 저장한다.
다음 대화에서는 전체 문서를 한 번에 훑어 성공, 실패, 결정, 인수인계, 지식, QA 상태를 즉시 복구한다.
