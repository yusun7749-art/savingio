# Savingio FRAMEWORK-FIRST DEVELOPMENT LOCK

최종 갱신: 2026-07-23 KST

## 사용자 최상위 지시

Savingio의 모든 작업은 그때그때 기능이나 페이지를 다시 작성하는 방식으로 진행하지 않는다.

반드시 먼저 공통 기본틀과 데이터 계약을 만든 뒤, 새 글·새 기능·새 분류는 정해진 정확한 위치에 데이터만 추가하는 방식으로 운영한다.

이 원칙은 검색 작업에만 한정되지 않으며 Savingio 전체 개발·콘텐츠·Navigation·내부링크·계산기·CMS·QA·배포에 공통 적용하는 최상위 LOCK이다.

## 폐기하는 방식

- 검색어 하나마다 예외 코드를 덧붙이는 방식
- 페이지마다 별도의 검색·분류 로직을 작성하는 방식
- 글을 추가할 때 HTML·JS·CSS를 매번 다시 만드는 방식
- 기능 오류가 날 때 기존 구조를 확인하지 않고 새 코드를 위에 덮어쓰는 방식
- 왼쪽 Navigation, 가운데 목록, 메인 검색이 서로 다른 데이터를 사용하는 방식
- 글 제목·카테고리·내부링크를 각각 수동으로 중복 관리하는 방식
- 구조를 정하지 않은 상태에서 글과 기능을 계속 늘리는 방식

## 영구 운영 원칙

### 1. 틀을 먼저 만든다

모든 작업은 아래 순서를 따른다.

요구사항 확인
→ 기존 공통틀과 데이터 위치 확인
→ 공통틀이 없으면 먼저 설계·구현
→ 정확한 삽입 위치 확정
→ 새 데이터만 추가
→ 공통 QA 실행
→ 실제 화면 검증

### 2. 구조와 데이터를 분리한다

공통 구조는 한 번만 만들고, 이후에는 내용만 교체하거나 추가한다.

예:

- 검색 엔진: 공통 JS 1개
- 검색 데이터: 공통 Search DB 1개
- 왼쪽 검색트리: 공통 UI 컴포넌트 1개
- 카테고리: 공통 분류 데이터 1개
- 내부링크: 공통 관계 데이터 1개
- 글 레이아웃: 승인된 Article DNA 1개
- 오른쪽 카드: 목적과 순서가 고정된 공통 템플릿 1개
- 계산기 연결: 공통 연결 규칙 1개

### 3. 새 글은 정확한 위치에 등록한다

새 글 작성 또는 기존 글 재작성 시 검색 코드와 Navigation 코드를 수정하지 않는다.

글 등록 데이터에 다음 항목을 정확히 입력한다.

- title
- url / slug
- large_category
- middle_category
- small_category
- primary_intent
- primary_keywords
- synonyms
- excluded_keywords
- related_articles
- related_calculators
- priority

이 데이터가 검색, 카테고리 버튼, 왼쪽 Navigation, 내부링크, 관련 글, 계산기 추천에 공통으로 사용되어야 한다.

### 4. 모든 화면은 단일 데이터 원본을 사용한다

다음 기능은 서로 다른 목록을 만들지 않고 동일한 Search/Navigation DB를 사용한다.

- 메인 검색창
- 정보센터 가운데 검색창
- 왼쪽 Brain 검색창
- 가운데 카테고리 버튼
- 왼쪽 카테고리 트리
- 관련 글
- 관련 계산기
- URL ?q= 검색

### 5. 글이 늘어나도 코드는 늘어나지 않아야 한다

글이 200개에서 500개, 1,000개로 증가해도 검색·분류·Navigation 코드를 글 수만큼 수정하지 않는다.

새 글은 데이터 레코드 1개를 추가하는 것으로 전체 시스템에 연결되어야 한다.

### 6. 검색 기본 원칙

검색은 관련 글을 무작위로 보여주는 기능이 아니라 사용자가 찾는 글을 가장 먼저 보여주는 기능이다.

고정 순위:

1. 제목 정확 일치
2. 제목이 검색어 또는 대표 동의어로 시작
3. 제목에 검색어 또는 대표 동의어 포함
4. 동일한 세부 검색 의도
5. 동일한 소분류
6. URL / slug 일치
7. 설명 일치
8. 본문 일치

본문에 단어가 있다는 이유만으로 제목 관련 글보다 앞설 수 없다.

### 7. 카테고리 버튼 기본 원칙

카테고리 버튼은 색상만 바꾸는 UI가 아니다.

클릭 시 공통 DB에서 해당 카테고리 전체 글을 불러오고, 현재 검색어와 충돌하지 않도록 검색 상태를 명확히 초기화하거나 결합 규칙에 따라 처리해야 한다.

### 8. 수정 전 정확한 위치를 찾는다

어떤 기능을 수정하든 먼저 다음을 확인한다.

- 실제 실행 파일
- 실제 로드되는 JS/CSS
- 공통 데이터 파일
- 기존 이벤트 핸들러
- 중복 로직
- 캐시 버전
- 데스크톱/모바일 분기

정확한 위치 확인 없이 새 코드를 추가하지 않는다.

### 9. 멱등성을 보장한다

같은 변환기·엔진·보정 스크립트를 여러 번 실행해도 요소가 중복 삽입되거나 구조가 깨지지 않아야 한다.

### 10. PASS 조건

다음 확인 전에는 PASS를 선언하지 않는다.

- GitHub main 반영
- 공통 데이터 연결 확인
- 메인 검색 테스트
- 가운데 검색 테스트
- 왼쪽 검색 테스트
- 카테고리 버튼 테스트
- 대표 검색어와 짧은 검색어 테스트
- 제목 우선순위 테스트
- 데스크톱/좁은 화면 레이아웃 테스트
- 실제 savingio.com 배포 화면 확인

## 현재 검색 작업에 적용할 구조

1. Search DB 확정
2. 전체 글의 대·중·소분류와 검색 의도 등록
3. Unified Search Engine 구현
4. Main / Center / Left Search Adapter 연결
5. Category Button Adapter 연결
6. Navigation Tree 연결
7. 내부링크·계산기 관계 연결
8. 검색 QA 세트 작성
9. Cloudflare 운영 화면 검증

## 사용자 강조 이유

사용자는 글을 오래 읽게 하는 것도 중요하지만, 먼저 찾을 수 있어야 글을 볼 수 있다고 명확히 지시했다.

현재 구조를 기본틀 없이 계속 임시 수정하면 글이 증가할수록 검색·분류·연결 작업이 더 어려워지므로, 지금 공통틀을 완성하는 것이 향후 전체 콘텐츠 확장의 선행 조건이다.

## 영구 금지

- 오류가 생길 때마다 검색어별 패치 추가
- 공통틀 없이 글 수만 늘리기
- 페이지별 독립 검색 구현
- 검증 없는 PASS
- 임시 결과를 완성된 기본틀로 보고
- 사용자가 강조한 FRAMEWORK-FIRST 원칙을 다음 대화에서 잊거나 초기화
